
# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано
Использовался датасет S12-hw-dataset.csv, прогнозировалась колонка 'target'.
Данные были разбиты на train / validation / test по времени (80/10/10%).
Сравнивались подходы: B1 (naive-last), B2 (moving-average), B3 (ridge-lag-features), R1 (GRU).
Лучшей моделью оказалась B3, выбранная по минимальному MAE на validation.

Датасет: S12-hw-dataset.csv  
Целевая колонка: target  
Частота временного ряда: дневная (по колонке 'date')  
Горизонт прогноза: 1

Диапазон дат: 2025-01-01 00:00:00 — 2025-06-29 23:00:00  
Есть ли дополнительные внешние признаки: Нет  
Комментарий: ряд содержит тренд и колебания, возможна сезонность. Присутствуют редкие выбросы и пропуски, прогноз сложный из-за непостоянного поведения ряда.

## 4. Temporal split и признаки

### 4.1 Разбиение по времени
Train / Validation / Test выполнен по времени.  
Доли: 80% / 10% / 10%  
Random split некорректен, так как приводит к утечке будущей информации в обучение.

### 4.2 Признаки для baseline-моделей
Лаги: lag_1, lag_7, lag_14  
Rolling-признаки: rolling_mean_7, rolling_std_7  
Календарные признаки: день недели  
Пропуски после lag/rolling удалялись (dropna)  
Масштабирование использовалось только для GRU и обучалось исключительно на train

## 5. Модели и эксперименты (B1, B2, B3, R1)

- **B1 (naive-last):** прогноз = последнее значение ряда  
- **B2 (moving-average):** прогноз = скользящее среднее с окном 7  
- **B3 (ridge-lag-features):** Ridge на lag, rolling и календарных признаках  
- **R1 (gru-forecast):** GRU на оконных последовательностях  

Основная метрика выбора лучшей модели: MAE на validation  
Дополнительные метрики: RMSE, MAPE  
Window_size для GRU: 14  
Batch size: 32  
Максимальное число эпох: 10  
Optimizer: Adam  
Learning rate: 1e-3  
Лучшая модель сохранялась через state_dict() в './artifacts/best_gru.pt'

## 7. Анализ
Наивный baseline оказался слабым, так как не учитывает сезонность и тренд.  
Moving average сгладил шум, но не учитывает лаговые зависимости.  
Ridge на признаках lag+rolling показала улучшение, особенно на среднесрочных колебаниях ряда.  
GRU захватила временную зависимость и помогла улучшить точность прогноза, особенно на нестабильных участках.  
Тренд и сезонность заметны и влияют на качество прогноза; пропуски были корректно обработаны.  
Утечки данных могли бы возникнуть через случайный split, масштабирование на всех данных, использование future lag.  
Выбор лучшей модели по validation принципиален, чтобы избежать bias от test.  
Наиболее показательными ошибками оказались резкие выбросы и смены тренда.  

## 8. Итоговый вывод
Базовым подходом для данного ряда можно считать Ridge с lag и rolling признаками.  
Простой baseline достаточен при стабильном ряде без резких колебаний.  
GRU полезна, когда важны долгосрочные зависимости и сложные паттерны.  
Главное — корректная temporal split валидация и предотвращение утечек будущей информации.
