# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- **Датасет**: использован `S12-hw-dataset.csv` – временной ряд с часовой периодичностью, прогнозировалась колонка `target`.
- **Разбиение**: организовано корректное временное разбиение на `train` (80%), `validation` (10%), `test` (10%) с сохранением хронологического порядка.
- **Подходы**: сравнивались `B1` (naive-last), `B2` (moving-average), `B3` (ridge на лаговых, rolling и календарных признаках), `R1` (GRU с оконным представлением).
- **Лучший подход**: `B3 (Ridge)` показал наилучший результат на валидации (MAE=5.2379) благодаря эффективной комбинации признаков, отражающих сезонность и тренд.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- **Разбиение**: выполнено строго по времени без перемешивания: первые 80% данных – train, следующие 10% – validation, последние 10% – test.
- **Интервалы**: Train: 3456 наблюдений (01.01.2025 – 20.04.2025), Validation: 432 наблюдения (20.04.2025 – 10.05.2025), Test: 432 наблюдения (10.05.2025 – 29.06.2025).
- **Почему random split некорректен**: временные ряды имеют автокорреляцию, random split нарушил бы временную структуру, создал бы утечку данных из будущего и привёл к нереалистично оптимистичной оценке качества.

### 4.2. Признаки для baseline-моделей

- **Лаги**: `lag_1`, `lag_7`, `lag_14`
- **Rolling-признаки**: `rolling_mean_7`, `rolling_std_7` (построены с использованием shift(1) для предотвращения утечки)
- **Календарные признаки**: час, день недели, день месяца, месяц, а также синус/косинус для цикличности (`hour_sin`, `hour_cos`, `dow_sin`, `dow_cos`)
- **Обработка пропусков**: строки с NaN (первые 14 наблюдений, для которых нет лагов) удалены после построения признаков
- **Масштабирование**: использован `StandardScaler`, обученный только на train, затем применён к validation и test

## 5. Модели и эксперименты (B1, B2, B3, R1)

- **B1 (naive-last)**: прогноз равен последнему известному значению. Простейший baseline.
- **B2 (moving-average)**: скользящее среднее с окном 24 часа (соответствует суточной сезонности ряда).
- **B3 (ridge-lag-features)**: линейная регрессия с L2-регуляризацией на 11 признаках (лаги, rolling, календарные). alpha=1.0.
- **R1 (gru-forecast)**: двухслойная GRU с hidden_dim=64, dropout=0.2, оконное представление window_size=24. Обучение 100 эпох с сохранением лучшей модели по валидационному loss.

**Дополнительно:**
- Основная метрика выбора лучшей модели: MAE на валидации
- Дополнительные метрики: RMSE, MAPE
- `window_size` для GRU: 24
- Batch size: 32
- Максимальное число эпох: 100
- Optimizer: Adam
- Learning rate: 0.001
- Сохранение лучшей модели: по минимальному val_loss

## 7. Анализ

- **Почему наивный baseline оказался слабым**: Naive baseline (MAE=6.26) не учитывает ни сезонность, ни тренд, что для ряда с выраженной суточной динамикой приводит к заметным ошибкам, особенно в периоды роста.
- **Что дали moving average и лаговые признаки**: MA с окном 24 часа оказался неожиданно слабым (MAE=13.38) – вероятно, из-за того, что сглаживание "размазывает" пики и не справляется с резкими изменениями. Лаговые признаки (особенно lag_1 с коэффициентом 13.86) оказались ключевыми для Ridge.
- **Почему Ridge на признаках сработала лучше простых baseline**: Ridge смогла эффективно комбинировать несколько источников информации: ближайшее прошлое (lag_1), сезонность (hour_cos, hour_sin) и долгосрочный тренд (month). Коэффициенты признаков показывают, что наибольший вклад вносит lag_1, затем часовая сезонность и месяц.
- **Почему GRU не помогла на данном ряде**: Несмотря на способность GRU к обучению долгосрочных зависимостей, для данного ряда с чёткой суточной сезонностью и небольшим трендом линейная модель с правильно подобранными признаками оказалась достаточной. GRU требовала больше данных и более тонкой настройки гиперпараметров, чем было проведено.
- **Насколько заметны тренд/сезонность**: Сезонность очень выражена (коэффициент hour_cos = -4.46), тренд также присутствует (коэффициент month = 2.65). Это подтверждает, что ряд нестационарен, но стационаризация через признаки оказалась эффективной.
- **Где в такой задаче могли бы появиться утечки данных**: Если бы rolling-признаки строились без shift(1) или если бы масштабирование выполнялось на всём датасете до split, произошла бы утечка из будущего. В работе все преобразования выполнялись корректно.
- **Почему выбор по validation, а не по test, принципиален**: Test должен использоваться только для финальной оценки. Если бы мы выбирали модель по test, мы бы "подглядывали" в будущее и получили бы смещённую оценку обобщающей способности.
- **Какие ошибки модели оказались наиболее показательными**: Ridge даёт ошибки в пиковых значениях (вечерние часы) и в периоды резких изменений. Это видно на графике финального прогноза.

## 8. Итоговый вывод

- **Какой подход взять как базовый**: Для данного ряда оптимальным базовым решением является Ridge на лаговых и календарных признаках – он прост, интерпретируем и показывает лучший результат.
- **В каком случае достаточно простого baseline на признаках**: Если ряд имеет чёткую сезонность и ограниченный объём данных, линейные модели с ручными признаками часто дают результат, сопоставимый или лучший, чем нейросети.
- **В каком случае имеет смысл переходить к GRU/другим нейросетям**: Переход к GRU имеет смысл при большем объёме данных, сложных нелинейных зависимостях или многомерных входных данных.
- **Что главное вы поняли про корректную валидацию временных рядов**: Главное в валидации временных рядов – никогда не нарушать временной порядок при разбиении, строить все признаки только из прошлого и использовать test только для финальной оценки.
